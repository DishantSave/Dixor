# Monotonic UUIDv7 Generation

> 📚 Documentation Navigation
>
> **Home:** [README.md](../../README.md)
> **Section:** [UUIDv7 Overview](./UUID7_Overview.md)

---

Dixor.Identity provides support for generating **monotonically increasing UUID version 7 identifiers**.

Monotonic UUID generation guarantees that identifiers generated within the same process are always produced in increasing order, even when multiple UUIDs are created during the same millisecond.

The public API exposes this functionality through:

```csharp
Guid Uuid7.NewMonotonicGuid()
```

Internally, this functionality is implemented by:

```csharp
Uuid7MonotonicGenerator
```

---

# What Does "Monotonic" Mean?

A sequence is said to be **monotonic** when every newly generated value is guaranteed to be greater than or equal to the previous value.

Example:

```text
1
2
3
4
5
```

This sequence is monotonic because every value increases.

The following sequence is **not** monotonic:

```text
1
3
2
5
4
```

---

# What Is The Problem With Standard UUIDv7?

Standard UUIDv7 already contains a timestamp.

Example:

```text
0197d8ef-f4c6-7a35-99e3-1d4a7e3c8e10
0197d8ef-f4c7-72af-bf71-6f3d9ef3f201
0197d8ef-f4c8-7b0d-925e-4dc10eaf92b2
```

These identifiers naturally sort by creation time.

However, timestamps are only accurate to:

```text
Milliseconds
```

Modern CPUs are extremely fast.

A single application may generate:

```text
Thousands or even millions of UUIDs
```

within the same millisecond.

Consider:

```csharp
for (int i = 0; i < 1000; i++)
{
    Guid id = Uuid7.NewGuid();
}
```

Many of these UUIDs may contain the exact same timestamp.

Because the remaining bits are random, their ordering becomes unpredictable.

Example:

```text
0197d8ef-f4c6-7a35-...
0197d8ef-f4c6-7f91-...
0197d8ef-f4c6-7210-...
```

Although generated sequentially, sorting may produce:

```text
0197d8ef-f4c6-7210-...
0197d8ef-f4c6-7a35-...
0197d8ef-f4c6-7f91-...
```

The original generation order has been lost.

---

# Why Monotonic UUIDv7 Exists

RFC 9562 recommends monotonic UUID generation for applications that require:

* Strict ordering.
* Sequential inserts.
* Better database locality.
* Reduced index fragmentation.
* Higher insertion throughput.

Monotonic UUIDs guarantee:

> Every newly generated UUID sorts after the previous UUID generated by the same process.

---

# Advantages of Monotonic UUIDs

## Strict Ordering

Generation order is preserved.

Example:

```csharp
Guid first = Uuid7.NewMonotonicGuid();
Guid second = Uuid7.NewMonotonicGuid();
Guid third = Uuid7.NewMonotonicGuid();
```

Guarantee:

```text
first < second < third
```

---

## Better Database Performance

Databases perform best when inserted keys are sequential.

Monotonic UUIDs improve:

* Insert speed.
* Page locality.
* Cache efficiency.
* Index maintenance.

Especially for:

* SQL Server
* PostgreSQL
* MySQL
* Oracle

---

## Reduced Index Fragmentation

Random identifiers scatter data pages.

Monotonic identifiers append new records near the end of indexes.

Benefits:

* Fewer page splits.
* Less fragmentation.
* Faster range scans.

---

## Improved Bulk Insert Performance

Applications performing:

* Batch imports
* ETL processing
* Data migration
* Event ingestion

benefit significantly from monotonic identifiers.

# Internal Architecture

Internally, the monotonic generator maintains two pieces of state.

```csharp
private static long _lastTimestamp;
private static ushort _counter;
```

---

# `_lastTimestamp`

Stores the timestamp associated with the most recently generated UUID.

Example:

```text
1751132765000
```

This value allows the generator to determine whether multiple UUIDs are being generated during the same millisecond.

---

# `_counter`

Stores an internal sequence number.

Example:

```text
0
1
2
3
4
```

Whenever multiple UUIDs are generated during the same millisecond, the counter increases.

This guarantees ordering.

---

# Generation Workflow

The monotonic generation process follows this workflow:

```text
Acquire Lock
      │
      ▼
Read Current Timestamp
      │
      ▼
Timestamp Equals Previous?
      │
 ┌────┴─────┐
 │          │
Yes         No
 │          │
 ▼          ▼
Increment   Reset Counter
Counter     Counter = 0
 │          Update Timestamp
 └────┬─────┘
      ▼
Generate Random Bits
      ▼
Embed Counter
      ▼
Create UUID
      ▼
Release Lock
      ▼
Return UUID
```

---

# Same Millisecond Behavior

Suppose the current timestamp is:

```text
1751132765000
```

The first UUID generated:

```text
Timestamp = 1751132765000
Counter   = 0
```

Second UUID:

```text
Timestamp = 1751132765000
Counter   = 1
```

Third UUID:

```text
Timestamp = 1751132765000
Counter   = 2
```

Fourth UUID:

```text
Timestamp = 1751132765000
Counter   = 3
```

All UUIDs share the same timestamp.

The counter ensures ordering.

---

# New Millisecond Behavior

Suppose the clock advances:

```text
1751132765000
```

becomes:

```text
1751132765001
```

The generator resets:

```text
_counter = 0
```

and updates:

```text
_lastTimestamp = currentTimestamp
```

The process then starts again.

---

# Counter Storage

The counter is stored using:

```csharp
ushort _counter;
```

A `ushort` occupies:

```text
16 bits
```

Maximum value:

```text
65,535
```

This allows:

```text
65,536 UUIDs
```

to be generated within a single millisecond while preserving ordering.

---

# Where Is The Counter Stored?

The counter is embedded into:

```text
Bytes 14 and 15
```

Implementation:

```csharp
bytes[14] = (byte)(_counter >> 8);
bytes[15] = (byte)_counter;
```

Example:

```text
Byte 14 -> High counter bits
Byte 15 -> Low counter bits
```

---

# UUID Layout

Monotonic UUID layout:

```text
0                   47 48   51
+---------------------+--------+
| Timestamp (48 bits) | Ver=7 |
+-----------------------------------------------+
| Random Bits + Variant + Counter (16 bits)   |
+-----------------------------------------------+
```

The final two bytes contain the monotonic sequence counter.

---

# Thread Safety

The monotonic generator is fully thread-safe.

Implementation:

```csharp
private static readonly Lock Lock = new();
```

Generation occurs inside:

```csharp
lock (Lock)
{
    ...
}
```

---

# Why Is Synchronization Required?

Without synchronization:

Two threads could execute simultaneously.

Example:

```text
Thread A -> Counter = 10
Thread B -> Counter = 10
```

Both UUIDs would receive the same counter value.

Ordering guarantees would be lost.

Synchronization ensures:

```text
Thread A -> Counter = 10
Thread B -> Counter = 11
```

The sequence remains monotonic.

---

# Concurrent Usage Example

The generator can safely be used from multiple threads.

```csharp
Parallel.For(
    0,
    1000,
    _ =>
    {
        Guid id =
            Uuid7.NewMonotonicGuid();
    });
```

No additional synchronization is required.

---

# Why Cryptographic Randomness Is Still Used

Monotonic UUIDs still contain:

```csharp
RandomNumberGenerator.Fill(...)
```

Benefits:

* Global uniqueness.
* High entropy.
* Collision resistance.
* Security.

The counter only affects ordering.

Randomness still guarantees uniqueness.

---

# Why Big Endian Is Used

Timestamp bytes are written using:

```text
Network Byte Order (Big Endian)
```

Implementation:

```csharp
bytes[0] = (byte)(currentTimestamp >> 40);
bytes[1] = (byte)(currentTimestamp >> 32);
...
```

Benefits:

* RFC compliance.
* Correct sorting behavior.
* Cross-platform compatibility.

---

# Why `stackalloc` Is Used

Implementation:

```csharp
Span<byte> bytes =
    stackalloc byte[16];
```

Benefits:

* No heap allocations.
* Reduced GC pressure.
* Better throughput.
* Improved performance.

Since UUIDs always require exactly sixteen bytes, stack allocation is extremely efficient.

Generate a monotonic UUID:

```csharp

    Uuid7.NewMonotonicGuid();
```


# Limitations

Monotonic guarantees are limited to:

```text
Single process
```

The generator cannot guarantee ordering across:

* Multiple processes.
* Multiple machines.
* Distributed clusters.

Example:

```text
Machine A
Machine B
```

may still produce interleaved UUIDs.

# Best Practices

## Use Monotonic UUIDs For Database Keys

Recommended:

```csharp
Guid id =
    Uuid7.NewMonotonicGuid();
```

for:

* Primary keys.
* Event streams.
* Ordered logs.

---

## Use Standard UUIDv7 When Strict Ordering Is Unnecessary

Monotonic generation introduces synchronization overhead.

If ordering is unnecessary:

```csharp
Uuid7.NewGuid()
```

may be preferable.

---

## Store UUIDs Using Native Database Types

Prefer:

```sql
uniqueidentifier
uuid
binary(16)
```

instead of strings.

---

# Summary

Monotonic UUID generation extends standard UUIDv7 by guaranteeing ordering for identifiers generated within the same process.

Benefits include:

* Strict ordering.
* Improved database locality.
* Reduced fragmentation.
* Better insert performance.
* Thread safety.
* RFC 9562 compliance.

Monotonic UUIDs are particularly valuable for high-throughput database applications and event-driven systems.
